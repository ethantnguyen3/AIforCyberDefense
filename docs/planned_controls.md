

 1. AI Anomaly Detection Pipeline (Detect)
The team plans to implement an AI-assisted anomaly detection pipeline integrated with SIEM data. This control will baseline normal user, endpoint, and network behavior, then identify suspicious deviations such as phishing-related login anomalies, unusual PowerShell execution, abnormal internal scanning, or multi-stage attack sequences. The purpose of this control is early detection of ransomware preparation, credential misuse, and lateral movement activity before major damage occurs.


 2. Automated SOAR Playbooks (Respond)
The team plans to develop SOAR playbooks that trigger when high-confidence alerts are generated by the detection pipeline. These playbooks will automate initial containment actions such as isolating compromised endpoints, disabling or suspending suspicious Active Directory accounts, and generating incident tickets for analyst review. This control is intended to reduce attacker dwell time and limit spread after a compromise is detected.

 
 3. Strict Default-Deny ACLs (Protect)
The team plans to harden the 3560 Layer 3 switch and related routing rules using a default-deny approach across trust boundaries. Only explicitly approved traffic types and paths will be allowed, especially between user networks, the Jump Host, application services, and backend systems. This control is designed to reduce lateral movement opportunities and prevent compromised lower-trust assets from reaching critical systems.


4. Jump Host Access Restrictions (Protect)
Because the Jump Host represents a high-trust administrative pathway, the team plans to limit access to only approved administrators, management subnets, and required protocols. Additional logging and monitoring will be applied to this path to detect attempts to bypass normal administrative controls.


 5. Application Code Remediation (Protect)
The team plans to remediate the application issues discovered during baseline testing, especially the state-saving and data persistence weaknesses that can cause records to disappear or become inconsistent. Input validation, storage handling, and API-side validation will be strengthened to reduce the risk of malformed record injection, data corruption, or abuse of client-side weaknesses.


 6. Validation and Baseline Comparison (Evaluate)
After controls are implemented, the team will compare the improved environment against the current baseline using logs, screenshots, prompts and outputs, and workflow testing. This will help measure whether the proposed controls meaningfully improve detection, containment, and resilience.
